# 회계기준-자문

K-IFRS 회계기준 질의에 **추적 가능한 문단번호 근거가 달린 회신 초안**을 만든다. 결론은 항상 근거 문단에 묶고, 근거가 없으면 단정하지 않는다. 최종 판단·서명은 담당 회계사·세무사가 한다.

> 이 스킬은 jaytax 지식베이스의 **단일 소스**를 근거로 동작한다. 번들된 `reference/` 의 세 파일이 그
> 단일 소스의 복사본이며, 권위 원본은 repo의 `accounting-standards/`·`prompts/` 다(아래 '동기화' 참조).

## 핵심 원칙 (반드시 지킬 것)
1. **모든 결론에 문단번호 근거를 붙인다.** 기준서번호·문단번호를 명시한다 (예: K-IFRS 제1115호 문단 31).
2. **정리본은 원문이 아니다.** 근거 소스(`reference/k-ifrs-1115.md`)는 *요지 정리본*이다. 인용 시 "(요지)"로 표기하고 **원문 대조 권고**를 붙인다. 요지를 큰따옴표 원문 인용처럼 쓰지 않는다.
3. **지어내지 않는다.** 근거를 못 찾은 항목은 `[확인 불가]`로 표기한다. 존재하지 않는 문단번호·링크를 생성하지 않는다.
4. **신뢰도 순서:** 원문(한국회계기준원·법령 MCP) > 지식 소스 정리본(`reference/`) > 웹(최신). 어긋나면 원문 우선, 불일치는 `[불일치]`로 명시.
5. **정보가 부족하면 되묻는다.** 추측으로 메우지 않는다 (아래 2단계·`reference/grounding-instructions.md` §2).
6. **자동 발송 금지.** 회신 초안까지만. 발송·서명은 사람이 한다.

## 근거 소스 (현재 범위)
- `reference/k-ifrs-1115.md` — K-IFRS 제1115호(고객과의 계약에서 생기는 수익) 요지 정리본, 127문단. 5단계 모형·계약원가·표시·공시·부록 A 용어정의·부록 B 적용지침·부록 C.
- 파일럿이라 **현재 1115호만** 적재돼 있다. 다른 기준서 질의는 `[확인 불가]`로 답하고, 단일 소스에 해당 기준서를 추가해야 함을 알린다.
- 정리본 전체가 컨텍스트에 들어가는 크기이므로, **파일을 직접 읽어** 관련 문단을 찾는다(파일럿 방식). 코퍼스가 커지면 Supabase 벡터검색(`match_accounting_standards` RPC)으로 전환한다 — 아래 '확장' 참조.

## 작업 순서

### 1단계 — 쟁점 도출
질의에서 회계 쟁점을 1~3개로 정리한다. 각 쟁점의 핵심 개념(예: 수익인식 시점 → 수행의무 이행·통제 이전·계약부채)과 적용 기준서를 식별한다.

### 2단계 — 근거 검색 (단일 소스)
1. `reference/k-ifrs-1115.md` 를 읽고, 쟁점에 해당하는 문단을 찾는다. 막연한 질의는 **상위개념·동의어로 다시** 훑는다 (예: "구독 선수금 인식시점" → 수행의무 이행 §31·§35, 계약부채 §106~107, 진행률 §39~45).
2. (가능하면) 찾은 문단번호로 **원문/법령 MCP**를 교차검증한다. 세법·상법이 섞이면 `korean-law` MCP로 조문도 함께 확인.
3. **충분성 판단** — `reference/grounding-instructions.md` §3 체크리스트로 답변에 필요한 정보가 갖춰졌는지 본다.

### 3단계 — 정보 부족 시 되묻기
사실관계·시점·수치가 불명확하면 **꼭 필요한 1~3개만** 되묻는다. 각 질문에 *왜 필요한지*를 한 줄로 붙인다. (`reference/grounding-instructions.md` §2의 표 참조.) 추측으로 답을 만들지 않는다.

### 4단계 — 회신 작성
`reference/email-template.md` 형식으로 작성한다 — 제목 → 사실관계/가정 → 검토 의견(문장 끝 각주 `[^n]`) → 결론 → 유의사항 → 하단 '근거'(문단번호+요지+원문대조 권고+내부참조). 모든 실질 주장에 각주를 달고 1:1로 매칭한다.

### 5단계 — 자기검증
- 모든 결론에 문단번호 근거가 붙었는가? 안 붙은 항목은 `[확인 불가]`인가?
- 인용한 문단번호가 실제 `reference/k-ifrs-1115.md` 에 존재하는가? (추측 인용 금지)
- 정리본 인용에 "(요지)"·원문 대조 권고가 붙었는가?
- 사실관계 가정을 회신 서두에 '가정:'으로 분리했는가?

## 출력 형식
`reference/email-template.md` 의 구조를 그대로 따른다. 예시·작성 규칙도 그 파일을 기준으로 한다. 본 스킬은 그 템플릿을 *채우는* 역할이며, 포맷을 임의로 바꾸지 않는다.

## 동기화 (단일 소스 ↔ 스킬 번들)
권위 원본은 repo의 `accounting-standards/`·`prompts/` 다. `reference/` 는 그 복사본이며 다음으로 갱신한다:

```bash
# repo 루트에서
npm run skill:sync        # accounting-standards/*.md + prompts/*.md → skill/reference/
```

배포: `skill/` 전체를 `~/.claude/skills/회계기준-자문/` 로 복사(또는 심볼릭 링크)하면 개인용 Claude Desktop 스킬로 동작한다. 단일 소스가 바뀌면 `skill:sync` 후 재배포한다.

## 확장 (코퍼스가 커질 때)
- 기준서가 늘어 정리본 전체가 컨텍스트에 안 들어가면, 파일 직접 읽기 대신 **Supabase 벡터검색**으로 전환한다:
  `scripts/standards/search.ts` 의 경로(질의 → OpenAI 임베딩 → `match_accounting_standards` RPC → 상위 N문단)를 스킬에서 호출. 이때 키(`OPENAI_API_KEY`·`SUPABASE_SECRET_KEY`)가 필요하다.
- 새 기준서 추가: `accounting-standards/k-ifrs-XXXX.md` 를 파싱 규약대로 작성 → `npm run std:parse:md` → (라이브면) `std:load` → `skill:sync`.

## 거버넌스
전문가 검토 전제 · 자동발송 금지(초안까지) · 결론마다 문단번호 근거 노출 · 정리본은 "(요지)"+원문 대조 권고 · 근거 미확보는 `[확인 불가]` · 문단번호·링크 지어내기 금지 · 고객 거래정보는 외부 웹 검색에 그대로 넣지 않음.
